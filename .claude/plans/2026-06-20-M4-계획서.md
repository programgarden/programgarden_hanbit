## ⚠️ 0. 범위와 최고위험 경계

### 0.1 M4 가 하는 일 — INV-1 의 첫 해제

M0~M3 동안 **실거래 주문 경로(국내주식·해외주식)는 코드에 존재하지 않았다**. 이것이 INV-1 의
구조적 강제였다:

- `order_registry.make_order_adapter()` 는 `overseas_futureoption`(paper) **하나만** 생성, 그 외 `LIVE_DISABLED`.
- `risk/gate.py:131` 게이트 step2 가 `intent.market != FUT → LIVE_DISABLED` 하드 거부(in-gate 이중방어).
- `order_service` 전체가 `MARKET_OVERSEAS_FUTUREOPTION` / `trading_mode="paper"` 로 하드코딩.
- 정적/행동 불변식 테스트(`test_readonly_invariant`·`test_no_live_facade`)가 **실거래 주문 facade·메서드 부재**를 컴파일·런타임 양쪽에서 박제.

**M4 는 이 모든 차단을 의도적으로, 통제 하에 해제한다.** 국내주식(`CSPAT00601` 등)·해외주식
(`COSAT00301` 등) 실주문 TR 이 app/ 에 처음 등장하고, `make_order_adapter` 가 LIVE 어댑터를 생성하며,
게이트가 LIVE 시장을 통과시킨다. **이 순간부터 전략·UI 의 주문은 실제 돈을 움직인다.**

### 0.2 설계 1순위 = 안전. "무엇이, 왜 열리는가"의 명문화

해제는 다중 직교 가드의 **모두** 통과를 전제로 한다(하나라도 막으면 실주문 불가):

| 가드 | 위치 | M4 동작 |
|---|---|---|
| **`HANBIT_ALLOW_LIVE`** (마스터 토글, 기본 `false`) | config + registry + 게이트 + 부트 | `false` 면 LIVE 버킷은 **영구 READ_ONLY** — 어댑터 생성·게이트·부트 ACTIVE 전부 차단. 실거래는 운영자가 명시적으로 `true` 로 켤 때만. |
| **per-bucket EngineState = ACTIVE** | `engine_state`(버킷별, §3) | LIVE 버킷이 부트 reconcile 성공 + allow_live 일 때만 ACTIVE. 아니면 주문 금지. |
| **소액 per-order 캡** | 게이트 §4(notional) | 국내 ≤ N KRW, 해외 ≤ M USD(§11, [R] 확정). 초과 REJECT. ⚠ **시장가/price=None/OrdPrc=0 은 notional 산정불가 → 캡 우회**(§17 L1-1) → LIVE 는 notional 미상 시 하드 REJECT. |
| **누적/일일 명목 한도** | 게이트 신규 단계(`daily_notional_used_krw`) | ⚠ **현 `bucket_notional_cap` 은 per-order 라 소액 다발 누적을 못 막는다**(§17 L1-3) → 트랜잭션 누적 캡 단계 신규 추가 필요. |
| **LIVE orderable 헤드룸** | 게이트 step7(§6) | LIVE 는 주문가능액(headroom) 미상 시 REJECT(`ORDERABLE_UNKNOWN_LIVE`)(§17 L1-9). `balances_snapshot` 라이브 배선이 **LIVE 활성 선행조건**(§13 M4d·§15). |
| **2단계 확인(confirm_token)** | API `/orders/quote→commit`(§10) | 사람 경로는 단발 토큰 강제. 전략 자동경로는 토큰 없이 게이트가 유일 방어선(§5.2 상위). |
| **첫주문 완충 가드** | 게이트/서비스(§10.2) | LIVE 첫 주문은 최소수량·단일종목·취소우선(롤백 가능)로 제한. |
| **킬스위치 L1/L2 + 일일손실 halt** | killswitch·halt(§9) | LIVE 버킷에도 동작(현재 L1 은 LIVE no-op — §9 에서 실동작화). |

**불변식은 약화가 아니라 진화한다(§12)**: `test_readonly_invariant` 의 "주식 주문 TR 부재" 단언은
"주식 주문 TR 은 **전용 어댑터 파일에만** 존재(스코프 봉인)" 로, `test_no_live_facade` 의 "LIVE 주문 0"
단언은 "**allow_live=false 면** LIVE 주문 0, **true + ACTIVE 면** 캡·confirm 통과분만" 으로 바뀐다.

### 0.3 모의(paper)로 사전검증 불가 — §2(Part I) 주의 재확인

국내·해외주식 실거래 경로는 **모의로 미러 검증되지 않는다**(라이브러리가 주식 paper 주문을 별도 제공하지
않음 — FUT 만 paper). 즉 M4 의 첫 실주문은 **진짜 첫 발사**다. 따라서:
- 코드·필드·성공판정은 §1 라이브러리 소스 검증 + §16 정정으로 확정하되,
- **실값(success rsp_cd·OrdNo 형식·체결 채널 필드·계좌 TR 호출건수)은 [L] 라이브 검증** 으로만 확정 가능 →
  Gate B(§14)에서 **소액·최소수량 1회 e2e** 로 검증한 뒤에만 전략 자동경로를 연다.

### 0.4 다음 절차(이 문서 작성 후)

③ **적대적 4-렌즈 검증**(워크플로우, M2/M3 절차) — 렌즈 후보: **(L1) 안전경계 누수**(allow_live/캡/2단계
우회 가능 경로), **(L2) 라이브러리 사실 오독**(§1 필드·성공판정·체결채널), **(L3) 버킷 격리 회귀**(per-bucket
engine state 도입이 paper M3 동작을 깨는가), **(L4) reconcile/멱등 정합**(list-all TR·OrdNo int·정정 새 OrdNo
승계). ④ 사용자 승인 → ⑤ 슬라이스(§13) 구현. **③은 멀티에이전트 비용이 크므로 사용자 opt-in 후 실행.**

### 1.5 어댑터 공통 사실
- **OrdNo 는 KR/OVS 둘 다 int** (FUT 는 str). `OrderAck.broker_ord_no:str` 이라 어댑터가 `str(OrdNo)` 정규화. ⚠ **OrdNo 일자 리셋 대비 = [L] 가 아니라 설계 결정사항(§17 L4-2)**: 현 `UNIQUE(account_id, broker_order_id)`·orphan 멱등키 `recon:{ordno}` 는 일자 차원이 없어, OrdNo 가 영업일마다 리셋되면 다음 거래일 충돌(ACCEPTED 전이 IntegrityError / orphan 미등록 / 어제 터미널 주문에 체결 오적용)이 난다. → **UNIQUE 를 `(account_id, trade_date, broker_order_id)` 또는 합성키 `'YYYYMMDD:ordno'`**, **orphan 멱등키 `recon:{ord_dt}:{ordno}`**, `get_order_by_broker` 도 일자 스코프. 일자경계 reconcile 회귀 테스트를 M4b/c DoD 에. (OrdNo 리셋 여부 자체는 여전히 [L] — Gate B e2e 로 확정, 단 키 설계는 선제 적용.)
- **정정은 새 OrdNo 발급**(KR/OVS 둘 다, FUT 동일) → `order_service.amend` 의 "성공 시 broker_order_id 갱신"(기존 코드) 그대로 적용.
- **취소에 수량 필요**: KR `CSPAT00801.OrdQty`, OVS `COSAT00301.OrdQty`(OrdPtnCode=08). 현재 `CancelRequest` 에 qty 없음 → **`CancelRequest.qty` 추가**(§4.3).
- 레이트리밋(소스 기본): KR 신규 10/s·정정·취소 3/s; OVS 신규·정정 10/s. 계좌 TR 호출건수는 [L](FUT 처럼 §8 직렬 큐 경유).

## 2. Config / 안전 토글

```python
# app/config.py 추가/변경
hanbit_allow_live: bool          # 既存(기본 False) — M4 마스터 토글. LIVE 버킷 ACTIVE 의 필요조건.
# engine_state 의도는 버킷별로 분리(§3):
#   - paper 의도 = 기존 HANBIT_ENGINE_STATE(READ_ONLY|PAPER_TRADING) 유지.
#   - live 의도  = hanbit_allow_live (별도 LIVE_TRADING 변수 불필요 — allow_live 가 곧 live 거래의도).
hanbit_live_per_order_cap_krw: int   # 국내 per-order 소액 캡(KRW). [R] §11.
hanbit_live_per_order_cap_usd: float # 해외 per-order 소액 캡(USD). [R] §11.
hanbit_live_first_order_guard: bool  # 첫주문 완충 가드 on/off(기본 True). §10.2.
```

- `HANBIT_ALLOW_LIVE` 가 **단일 마스터**: false → registry 거부 + 게이트 거부 + 부트가 LIVE 버킷을 READ_ONLY 고정. 한 곳만 꺼도 안전(다층).

## 3. 아키텍처 핵심 결정 — **per-bucket EngineState**

### 3.1 문제
현재 `EngineState` 는 **프로세스 단일 인스턴스**(`order_service._engine`). 부트(`boot.py`)는
`BUCKET_PAPER` 만 reconcile 하고 단일 상태를 ACTIVE/READ_ONLY 로 전이. M4 는 **live 버킷이 추가**되고,
두 버킷은 **독립적으로** 부팅·실패·차단될 수 있다(예: live reconcile 실패해도 paper 는 계속 거래; allow_live=false 면 live 만 영구 READ_ONLY).

### 3.2 결정: 버킷별 EngineState (단일 글로벌 폐기)
```python
# 채택: order_service 가 버킷→EngineState 맵을 소유
self._engines: dict[str, EngineState] = {
    BUCKET_PAPER: EngineState.from_paper_config(settings),   # 기존 동작 보존
    BUCKET_LIVE:  EngineState.live_initial(settings),         # allow_live=false → READ_ONLY 고정
}
def engine_for(self, bucket: str | None) -> EngineState:     # 버킷→엔진 단일 접근자
    eng = self._engines.get(bucket) if bucket else None      # 미지 시장(bucket None)=거부
    if eng is None:
        raise OrderError("LIVE_DISABLED", f"no engine for bucket {bucket!r}")
    return eng
# 게이트 step0 / _get_mutable / 실시간 writer: engine_for(bucket_of(market))
```
- **게이트 step0**(`gate.py:94`)·**`_get_mutable`**(`order_service.py:450`)·**`boot_engine`**·**실시간 writer(§8)** 모두 `engine_for(bucket_of(market))` 로 해당 버킷 상태를 읽는다. ⚠ **미지 시장 → `bucket_of`=None → `engine_for` 가 엔진 선택 전에 `LIVE_DISABLED` 거부**(§17 L3-4: `engines[None]` KeyError 방지, 기존 `if bucket else` 관용구와 정합). 실시간 writer 는 `engine_for(BUCKET_LIVE)`(KR/OVS)·`engine_for(BUCKET_PAPER)`(FUT)를 주입받아 **버킷별 ACTIVE** 를 본다(§17 L3-8: 단일 paper 엔진 교차결선 방지). 호환 property `engine`(단수)=`engines[BUCKET_PAPER]` 는 paper-only deprecated.
- `boot_engine` 은 **버킷별로 반복하되 각 버킷을 독립 try/except 로 감싼다**(§17 L3-11): paper(CIDBQ*) + live(CSPAQ13700/COSAQ00102/COSOQ00201). **한 버킷의 reconcile/동기화 예외가 다른 버킷 ACTIVE 전이에 영향 0**(예: live reconcile 예외 → live READ_ONLY 유지 ∧ paper 정상 ACTIVE). live 버킷 ACTIVE 조건 = **`allow_live=true` AND live reconcile 성공 AND 포지션 동기화 성공**. allow_live=false 면 reconcile 자체를 skip 하고 READ_ONLY 고정.
- `/system/health` 는 기존 `engine_state`(=paper 값) **하위호환 유지** + `engine_states:{paper,live}` 맵 **신규 추가**(§17 L3-9: 기존 string-equality assert **2곳**[`test_orders_api`·`test_portfolio_api`] 무파괴; `test_reconcile_boot` 의 engine_state 단언은 `BootReport.engine_state` 라 health 응답과 무관). ⚠ **WS 는 현재 engine_state 미노출** → `engine_states` 맵을 신규로만 추가(하위호환 대상 없음). `/system/modes` 는 engine_state 무관(모드 매트릭스만) — 이 노출 대상에서 분리.

### 3.3 대안(기각): 단일 글로벌 + allow_live 만으로 live 게이팅
- 더 작은 변경이나 **버킷 격리 위반**: live reconcile 실패가 paper 거래를 동결(또는 그 반대). M3 버킷 격리 원칙(Part III §3)과 모순 → 기각. (단 구현 복잡도상 §13 에서 **M4a 는 단일→맵 리팩터만** 먼저 green 으로 분리.)

per-bucket 도입은 **paper 동작 무변**이 DoD(검증 렌즈 L3). 기존 `engine`(단수) 접근자는

## 7. reconcile 일반화 — list-all + 버킷별

FUT reconcile(종목별 CIDBQ02400 루프)과 달리 **KR/OVS 는 list-all**(§16 C7):
```
reconcile(bucket, scope):                               # boot 는 boot:paper + boot:live 2회 호출(§17 L3-5)
  prio = priority_of(scope)                              # kill > boot > routine
  for market in markets_of(bucket):                      # live=(KR,OVS) / paper=(FUT,)
    adapter = make_order_adapter(market, ..., allow_live=...)
    acct = get_account_id(market)                        # 시장별 계좌 앵커(교차 금지)
    broker_rows = paginate(market, adapter, prio)        # tr_cont+cursor 2층·페이지당 submit 1회(락 밖)
    # OrdNo 매칭 / 체결적용 / 소멸판정(fail-safe) / orphan(OrgOrdNo 체인) = FUT 로직 + 아래 보정
    positions = await tr.submit(bucket, prio, adapter.get_positions)   # KR=CSPAQ12300 / OVS=COSOQ00201
    for p in positions:                                  # 권위 writer 는 이 market/bucket/currency 로만
      upsert_position_authority(acct, inst_of(market, p.symbol),
                                bucket=bucket, market=market, currency=p.currency)  # 교차 upsert 금지
```
- **이중 연속 메커니즘 필수**(§17 L4-1): 라이브러리는 **(1) 응답헤더 `tr_cont=='Y'` 루프**(`tr_helpers.occurs_req`)와 **(2) 애플리케이션 `SrtOrdNo`/`SrtOrdNo2` cursor** 두 층이 있다. **둘 다** 다뤄야 한다 — 각 cursor 페이지마다 tr_cont 후속 프레임을 모두 수집한 뒤 최소 OrdNo 로 cursor 진전. ⚠ **cursor 산술이 시장마다 다름**(§17 L2-4): **KR=최소 OrdNo − 1, OVS=최소 OrdNo**(공유 paginate 가 한 규칙 가정 금지).
- **fail-safe(소멸 판정 보류, §17 L4-1)**: 페이지네이션이 정상 종료(빈 페이지/cursor 무진전)에 도달 못 했거나 max_pages 가드에 걸리면, 그 reconcile 사이클은 **미스매치→CANCELED/EXPIRED 전이를 skip**(소멸 판정 보류)하고 `position_sync_ok` 처럼 incomplete 플래그로 호출자에 알린다. 종료조건 = `빈 block3 OR cursor 무진전 OR max_pages`. (미완 페이지가 살아있는 working 주문을 잘못 소멸 처리하는 실거래 사고 차단.)
- **정정 후 orphan 오등록 차단(§17 L4-3)**: KR/OVS 미체결 TR 은 **체결내역(ExecYn 기본=전체)** 이라 정정 직후 **구 OrdNo + 신 OrdNo 가 같은 스냅샷에 공존**한다. orphan 등록 전 broker row 의 `OrgOrdNo`(부모 OrdNo)가 DB 의 알려진 broker_order_id(현재/과거 승계 이력)에 매칭되면 **orphan 이 아니라 정정 후속**으로 간주(OrgOrdNo 체인 인식). FUT OpenOrder 는 이미 `org_ord_no` 보유. 회귀 테스트(`test_live_reconcile` 정정-후-orphan). 가능하면 `get_open_orders` 의 `ExecYn` 을 미체결-only 로 설정해 노출 자체 축소.
- **계좌 TR 은 §8 직렬 큐(tr_queue) 경유 + 락순서 불변식(§17 L4-6)**: 페이지네이션 각 페이지의 `get_open_orders`/`get_positions` submit 은 **반드시 order 락 밖에서**(FUT 패턴 동일 — 락은 매칭/체결적용/전이 구간만). **페이지당 submit 1회 → `max_pages` 가 곧 호출건수 상한**. 우선순위 boot=BOOT/routine=ROUTINE/kill=KILL. ⚠ live 버킷 큐가 KR+OVS 를 한 키로 직렬화하면 기아 가능 → **큐 sub-key `bucket:market` 분리 여부 [결정사항]**(aging 존재로 기본은 bucket 키 허용). §9 L1(live) precancel reconcile 도 락 밖 호출.
- **권위 writer 시장별 라우팅(§17 L3-5)**: reconcile 의 `ensure_instrument`/`upsert_position_authority`/`get_account_id` 는 **루프의 market·bucket·currency 만** 소비(위 의사코드) — 버킷 교차 upsert 금지. (instrument_id 가 시장별로 달라 행 충돌은 구조적으로 없으나 bucket/currency 컬럼 오기록 방지를 불변식으로 명문화.) 집계기 보강(가격/eval/fx)은 M3 그대로. OVS 는 잔고 TR 에 FX(`BaseXchrat`)·평가손익 내장 → 집계기 입력으로 활용.

## 8. 실시간 체결 (SC1/AS1 스캐폴드)

M3b TC2/TC3 스캐폴드(`realtime_future.py`)와 **동형**:
- 신규 `app/adapters/realtime_korea.py`(SC1)·`realtime_overseas.py`(AS1). 순수 정규화 `sc1_to_fill`/`as1_to_fill` + `start()/stop()` 구독.
- **멱등키**: KR `event_seq='sc:'+ordno+':'+execno`, OVS `'as:'+sOrdNo+':'+sExecNO`. 체결 판정: KR `ordxctptncode=='11'`, **OVS `sExecQty>0`(11 코드 없음)**. ⚠ **교차-origin 이중계상 주의(§17 L4-5)**: `sc:`/`as:` prefix 는 reconcile `recon:` prefix 와 달라, 동일 물리체결(같은 execno)이 양쪽에서 오면 fills UNIQUE`(order_id,event_seq)` 가 dedup 못 해 **2행 적재**. 현 보호 = **flag-off 기본 + reconcile=체결 권위**(둘 다 ON 금지). 장기 옵션: origin 무관 통일키 `exec:{ordno}:{execno}`.
- 적재는 reconcile 과 **동일 단일 writer 경로** `repo.apply_fill(trigger='sc'/'as')` — OrderLocks 공유. 미등록 OrdNo 는 reconcile orphan 위임(reconcile=체결 권위).
- `writer_enabled = hanbit_realtime_fills AND 해당버킷 EngineState.ACTIVE` (M3b 규칙 동일). ⚠ 해당버킷 엔진은 **`engine_for(BUCKET_LIVE)`(KR/OVS)·`engine_for(BUCKET_PAPER)`(FUT)를 주입**받아 본다(§17 L3-8 — 단일 paper 엔진 교차결선 금지). main.py lifespan 이 시장별 source 에 올바른 엔진 주입.
- 거부 ack(`'14'`/`sRjtRsn`)은 관측 dict(상태머신 REJECTED 전이는 reconcile/place ack 권위 유지). `as1_to_fill` 은 **`sExecQty<=0` 이면 None 반환**(거부 ack 를 체결로 오적재 차단, §17 L4-5).
- **DoD 아님(스캐폴드, flag off 기본)** — 라이브 값 검증은 [L]. reconcile 이 체결 권위.

## 9. 킬스위치 / 청산 — LIVE 버킷 실동작화

현재 `killswitch.py` L1 은 **LIVE 버킷 = 명시적 no-op-with-warning**(M3b §9, INV-1 보호). M4 는 이를 실동작으로:
- **L1(미체결 일괄취소)**: LIVE 버킷도 `cancel_all_open(bucket=live)` 실행 — 위험감축 lane(engine-state 우회) 재사용. 취소는 멱등·노출감소라 allow_live 와 무관하게 **항상 허용**(오히려 끄려면 켜져야 함). quarantine 노출(raw cancel-by-OrdNo) 동일.
- **L2(전 포지션 청산)**: LIVE 도 reduce-only EXIT 발사. ⚠ **현 `flatten_all_positions`/`resume_pending_flatten` 의 `assert bucket=='paper'` 제거 → `buckets_of` 라우팅**(§17 L3-7), `engage_level2` 의 flatten 루프를 대상 버킷 전체(global=paper+live)로 확장, `_flatten_one` 의 market/거래소/주문유형 시장별 분기. ⚠ 시장가 청산 정책([R] §11): KR/OVS 시장가(`OrdprcPtnCode="03"`) 수용 여부 [L], 미수용 시 지정가+슬리피지 fallback. **단 청산 명목도 §6 step4'(notional-필수) 통과**(reduce-only 라도 명목 산정·기록). 장마감 `pending_flatten` 큐 동일.
- `halt.py`: `bucket_state`/`states_snapshot`/`normalize_scope` 가 이미 `BUCKET_LIVE` 포함(무변). `normalize_scope` 에 KR/OVS 시장 키 추가.
- ⚠ **LIVE no-op 제거는 §12 행동 불변식(`test_no_live_facade`)을 의도적으로 갱신** — "killswitch L1(LIVE)→어댑터 미진입" 단언이 "allow_live=true 면 취소 진입 / false 면 미진입" 으로 바뀐다.

## 10. 사람 경로 2단계 확인 + 첫주문 완충 가드

### 10.1 quote→commit (Part I §5.1)
- `POST /orders/quote`: 게이트 preview(주문 미발사) + **confirm_token**(짧은 TTL·단발·서버발급) 발급 + est(notional/fee/post_trade_exposure/currency/mode). LIVE 면 빨강 헤더 메타.
- `POST /orders/commit{confirm_token}`: 토큰 검증(one-time·미만료) → **재검증**(모드·캡·누적명목·리스크·킬스위치, 시간차 변동 대비) → place. ⚠ **현 killswitch L2 의 confirm_token 패턴은 무상태(stateless)**(아무 비빈 토큰이나 통과 — TTL/일회성/intent 바인딩 없음, §17 L1-10) → M4 는 **서버측 토큰 저장(짧은 TTL·one-time 소모·intent 해시 바인딩)** 을 신규로 구현해야 'one-time·미만료'가 성립. **`commit` 이 토큰 없으면 LIVE 거부**(quote 없이 commit 직접호출 우회 차단) — M4d DoD 에 "quote 없이 commit→거부" 테스트.
- 전략 자동경로는 토큰 없음 — 게이트가 유일 방어선(Part I §5.2, 특히 §6 step4'' 누적캡). LIVE 전략 기본 한도 보수적.

### 10.2 첫주문 완충 가드(로드맵 M4 DoD "live 첫주문 완충 가드")
`hanbit_live_first_order_guard=true` 동안 LIVE 버킷 주문에:
- **최소수량 강제**(1주/최소 board lot), **단일종목 제한**(동시 1종목), **취소우선**(미체결 즉시 취소 경로 우선 노출). Gate B(§14) e2e 1회 성공 후 운영자가 해제.
